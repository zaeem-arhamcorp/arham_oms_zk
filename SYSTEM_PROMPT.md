# Arham OMS - System Prompt for AI Assistance

Arham OMS is a production-grade Flutter-based Order Management & Employee Tracking System for field operations (base URL: https://apidev.arhamcorp.in/api, Android API 36+, Firebase Crashlytics enabled). The app handles employee location tracking, route management, order placement, inventory management, and punch in/out operations with full offline capability. It uses Provider for state management, GetX for reactive updates, and SQLite for offline caching with critical constraints like UNIQUE(item_cd, party_cd) on cart items. Key modules include location_service.dart for GPS/punch tracking (optimistic loading, blocks punch offline), product_page.dart and shoppingCartPage.dart for order management (optimistic delete with "will sync online"), route_map_view.dart for map tracking with multi-stage fetching, reimbursement_approvals_view.dart for approval workflows, adduserScreen.dart for user creation, services.dart for centralized API calls (Bearer token auth), and app_snack_bar.dart for notifications. All API headers require: Authorization: Bearer $token, x-app-type: oms, Content-Type: application/json.

CRITICAL PATTERN - Mounted Guard: Every async callback accessing context or setState must check if(!mounted)return; before setState or context access. This prevents "Null check operator used on a null value" crashes when widgets dispose before async callbacks complete. Applied in: product_page.dart (6+ places), shoppingCartPage.dart (inline), route_map_view.dart (8 places - pagination, users, trips), reimbursement_approvals_view.dart (5 places), services.dart (Provider.of). Other patterns: Optimistic Loading - try API immediately with no pre-flight checks, fall back to offline on failure (delete cart items, location updates). TextEditingController Cleanup - store late VoidCallback listeners, remove in dispose() to prevent setState after disposal. Bounds Checking - validate index bounds in ListView.builder when list is filtered. GetX Error Handling - wrap Get operations in try-catch (not always initialized). Android Version Parsing - extract major version via version.split('.').first before int.parse(). All logging uses prefixes like [LocationService], [ORDER], [CART] for filtering. When adding features: add mounted guards to all async callbacks, use optimistic loading without network checks, store controller listeners and remove in dispose(), validate list bounds, wrap GetX in try-catch, handle edge cases in parsing. Offline operations allowed: browsing products, viewing history, settings. Not allowed: punch in/out, API-dependent operations. Performance: 10s API timeout typical, users endpoint returns 20 items/page, workmanager batches background tracking. Common errors: null pointer at setState → missing mounted guard; LateInitializationError → wrap GetX in try-catch; RangeError in ListView → add bounds check; setState after dispose → store and remove listener; FormatException in version → use split('.').first.
